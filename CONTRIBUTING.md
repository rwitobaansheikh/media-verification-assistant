# Contributing

## Workflow

1. Find or create an issue.
2. Comment that you want to work on it.
3. Wait for maintainer acknowledgement if the issue is large or touches shared architecture.
4. Create a feature branch.
5. Keep PRs small and link the issue.
6. Use the PR template and call out tests, docs, and limitations.

## Branches

Use short descriptive branch names:

- `docs/project-charter`
- `research/dataset-inventory`
- `frontend/report-ui`
- `backend/analysis-api`
- `ml/model-adapter`

Do not push directly to `main`.

## Research Contributions

Research PRs should include:

- sources with links;
- retrieval date when relevant;
- license/access notes for datasets or model repos;
- clear distinction between source claims and your interpretation;
- recommended follow-up issues.

Use [docs/research/research-brief-template.md](docs/research/research-brief-template.md).

## Privacy and Safety

Do not commit:

- raw meeting recordings or transcripts;
- private community exports;
- uploaded user media;
- credentials, tokens, or API keys;
- large datasets;
- model weights;
- copyrighted media samples unless the license is clear.

If a sample file is needed, use a tiny, clearly licensed fixture and document its source.

## Review Expectations

Maintainers should review for:

- correctness;
- scope control;
- privacy and safety risk;
- source quality for research;
- clear documentation;
- test or verification evidence where appropriate.
